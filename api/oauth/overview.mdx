로켓펀치 API는 두 가지 인증 모델을 지원한다. 공개 플랫폼 데이터에 대한 서버 사이드 접근을 위한 App Key 인증과, 특정 로켓펀치 사용자를 대신해 데이터에 접근하기 위한 OAuth 2.0이다. 애플리케이션이 무엇을 해야 하는지에 따라 올바른 모델을 선택한다. 대부분의 연동은 App Key 인증으로 시작하고, 사용자별 기능이 필요할 때만 OAuth를 추가한다.

## 언제 OAuth를 사용하는가

두 인증 방식 모두 동일한 기반 API를 사용하지만, 목적이 다르다. 활용 사례에 맞는 것을 선택한다.

| 활용 사례 | 방식 |
| --- | --- |
| 플랫폼에서 채용 공고 검색 | App Key |
| 앱에 기업 프로필 표시 | App Key |
| 공개 네트워킹 이벤트 조회 | App Key |
| 로그인한 사용자의 로켓펀치 프로필 접근 | OAuth 2.0 |
| 사용자가 저장한 채용 공고나 선호 읽기 | OAuth 2.0 |
| 로켓펀치 내에서 사용자를 대신해 행동 | OAuth 2.0 |

**App Key 인증**은 서버가 특정 사용자를 식별하거나 대신하지 않고 공개 로켓펀치 데이터(채용 공고, 기업, 이벤트)를 호출할 때 적합하다. 매 요청에 `X-RP-API-Key` 헤더를 포함하며, 응답은 공개적으로 이용 가능한 플랫폼 데이터를 반영한다.

**OAuth 2.0**은 애플리케이션이 로켓펀치 사용자에게 개인 데이터를 읽거나 상호작용할 권한을 요청해야 할 때 적합하다. 사용자가 로켓펀치에 로그인하고, 앱이 요청하는 내용을 검토한 뒤 접근을 승인한다. 그러면 해당 사용자로 범위가 지정된 토큰을 받아 이후 API 호출에 사용한다.

<Note>
  채용 공고, 기업 프로필, 이벤트 같은 공개 데이터만 필요하다면 OAuth가 필요 없다. 대신 [App Key 인증](/api/authentication)을 설정하고 OAuth 흐름은 건너뛴다.
</Note>

## OAuth 2.0 Authorization Code Flow

로켓펀치는 웹 및 서버 사이드 애플리케이션을 위한 업계 표준 방식인 **OAuth 2.0 Authorization Code Flow**를 구현한다. 다음은 단계에 대한 상위 수준 개요다:

1. **리디렉션**: 앱이 사용자를 로켓펀치의 인증 엔드포인트로 리디렉션하며, `client_id`, 요청 스코프, 동의 후 로켓펀치가 사용자를 다시 보낼 `redirect_uri`를 포함한다.
2. **동의**: 사용자가 로켓펀치 로그인 및 권한 화면을 본다. 승인하면 로켓펀치가 단기 `code` 파라미터와 함께 사용자를 `redirect_uri`로 리디렉션한다.
3. **토큰 교환**: 서버가 로켓펀치 토큰 엔드포인트로 POST 요청을 보내 `code`를 `access_token`(및 선택적으로 `refresh_token`)으로 교환한다.
4. **API 호출**: 앱이 사용자별 API 요청에 `Authorization: Bearer` 헤더로 `access_token`을 포함한다.

전체 구현과 코드 예시는 [Auth Flow 가이드](/api/oauth/flow)를 참고한다.

## OAuth 설정하기

흐름을 구현하기 전에 로켓펀치 개발자 포털에서 애플리케이션을 등록해야 한다.

<Steps>
  <Step title="앱 만들기">
    [developers.rocketpunch.com/apps/new](https://developers.rocketpunch.com/apps/new)로 이동해 애플리케이션 이름과 설명을 입력한다. 이 정보는 동의 화면에서 사용자에게 표시된다.
  </Step>
  <Step title="리디렉션 URI 추가">
    사용자가 동의를 부여(또는 거부)한 후 로켓펀치가 리디렉션할 콜백 URL을 입력한다. 이는 인증 요청에 사용하는 `redirect_uri`와 프로토콜, 도메인, 경로까지 정확히 일치해야 한다.

    ```text
    https://yourapp.com/auth/rocketpunch/callback
    ```
  </Step>
  <Step title="자격 증명 복사">
    앱을 저장하면 포털이 **Client ID**와 **Client Secret**을 표시한다. Client Secret을 안전하게 보관하고 비밀번호처럼 다룬다. 토큰 교환 단계를 구현할 때 두 값이 모두 필요하다.
  </Step>
</Steps>

<Warning>
  Client Secret은 클라이언트 사이드 코드, 모바일 앱 바이너리, 공개 저장소에 절대 나타나서는 안 된다. 토큰 교환은 항상 서버에서 이뤄져야 한다.
</Warning>

## 스코프

스코프는 애플리케이션이 접근할 수 있는 사용자 데이터를 정확히 정의한다. 사용자를 인증 엔드포인트로 리디렉션할 때 필요한 권한을 나열하는 `scope` 파라미터를 포함한다. 사용자는 승인 전 동의 화면에서 이 권한을 본다.

앱이 정말로 필요한 스코프만 요청한다. 불필요한 권한을 요청하면 사용자 신뢰가 떨어지고 승인율이 낮아진다. 이용 가능한 스코프는 [Auth Flow 가이드](/api/oauth/flow)에 나열되어 있다.
