레이트 리밋은 모든 개발자가 일관되고 안정적인 성능을 얻도록 로켓펀치 API를 과도한 트래픽으로부터 보호한다. 모든 API 키는 롤링 시간 창에 걸쳐 측정되는 요청 할당량의 적용을 받는다. 앱이 할당량을 초과하면 API는 창이 리셋될 때까지 `429 Too Many Requests` 응답을 반환한다. 처음부터 재시도 로직과 캐싱을 연동에 구축하면 이런 한도가 사용자에게 영향을 주는 것을 막을 수 있다.

<Note>
  앱 등급별 구체적인 레이트 리밋 값은 [developers.rocketpunch.com](https://developers.rocketpunch.com)의 개발자 콘솔에 표시된다. 활용 사례에 더 높은 할당량이 필요하면 [support@rocketpunch.com](mailto:support@rocketpunch.com)으로 문의해 상향을 논의한다.
</Note>

## 레이트 리밋 헤더

모든 API 응답에는 할당량의 현재 상태를 알려주는 헤더가 포함된다. 이 헤더를 읽어 사용량을 모니터링하고 예기치 않게 한도에 도달하는 것을 피한다.

| 헤더 | 설명 |
| --- | --- |
| `X-RateLimit-Limit` | 현재 창에서 허용되는 총 요청 수 |
| `X-RateLimit-Remaining` | 한도에 도달하기 전 남은 요청 수 |
| `X-RateLimit-Reset` | 현재 창이 리셋되고 할당량이 복원되는 Unix 타임스탬프(UTC) |

성공한 응답에서 이 헤더가 어떻게 보이는지에 대한 예시는 다음과 같다:

```http
HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1718150400
Content-Type: application/json
```

리셋까지 남은 초는 다음과 같이 계산할 수 있다:

```javascript
const resetTime = parseInt(response.headers.get("X-RateLimit-Reset"), 10);
const secondsUntilReset = resetTime - Math.floor(Date.now() / 1000);
```

## 레이트 리밋 오류 처리

앱이 할당량을 소진하면 API는 `429 Too Many Requests`와 함께 재시도 전 대기할 초를 나타내는 `Retry-After` 헤더로 응답한다.

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1718150400
Content-Type: application/json

{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many requests. Please retry after the reset window.",
    "status": 429
  }
}
```

레이트 리밋을 처리하는 가장 견고한 패턴은 **지터를 곁들인 지수 백오프**다. 첫 실패 후 기본 간격만큼 대기하고, 이후 실패마다 두 배로 늘리며, 여러 인스턴스의 동기화된 재시도를 방지하기 위해 작은 무작위 오프셋을 더한다.

<CodeGroup>

```javascript Node.js
async function fetchWithRetry(url, options, maxRetries = 4) {
  let attempt = 0;

  while (attempt <= maxRetries) {
    const response = await fetch(url, options);

    if (response.status !== 429) {
      return response; // 성공 또는 레이트 리밋이 아닌 오류
    }

    if (attempt === maxRetries) {
      throw new Error(`Rate limit exceeded after ${maxRetries} retries`);
    }

    // Retry-After 헤더를 읽고, 없으면 지수 백오프로 대체
    const retryAfter = response.headers.get("Retry-After");
    const baseDelay = retryAfter
      ? parseInt(retryAfter, 10) * 1000
      : Math.pow(2, attempt) * 1000;

    // 동기화된 재시도를 피하기 위해 지터(±20%) 추가
    const jitter = baseDelay * 0.2 * (Math.random() * 2 - 1);
    const delay = Math.round(baseDelay + jitter);

    console.log(`Rate limited. Retrying in ${delay}ms (attempt ${attempt + 1})`);
    await new Promise((resolve) => setTimeout(resolve, delay));
    attempt++;
  }
}

// 사용법
const response = await fetchWithRetry(
  "https://openapi.rocketpunch.com/api/v1/jobs",
  {
    headers: {
      "X-RP-API-Key": process.env.RP_API_KEY,
    },
  }
);
const data = await response.json();
```

```python Python
import time
import random
import requests

def fetch_with_retry(url, headers, max_retries=4):
    attempt = 0

    while attempt <= max_retries:
        response = requests.get(url, headers=headers)

        if response.status_code != 429:
            return response  # 성공 또는 레이트 리밋이 아닌 오류

        if attempt == max_retries:
            raise Exception(f"Rate limit exceeded after {max_retries} retries")

        # Retry-After 헤더를 읽고, 없으면 지수 백오프로 대체
        retry_after = response.headers.get("Retry-After")
        base_delay = float(retry_after) if retry_after else (2 ** attempt)

        # 동기화된 재시도를 피하기 위해 지터(±20%) 추가
        jitter = base_delay * 0.2 * (random.random() * 2 - 1)
        delay = base_delay + jitter

        print(f"Rate limited. Retrying in {delay:.1f}s (attempt {attempt + 1})")
        time.sleep(delay)
        attempt += 1

# 사용법
response = fetch_with_retry(
    "https://openapi.rocketpunch.com/api/v1/jobs",
    headers={"X-RP-API-Key": os.environ["RP_API_KEY"]},
)
data = response.json()
```

</CodeGroup>

## 모범 사례

정상 운영에서 레이트 리밋을 충분히 여유 있게 유지하려면 다음 사례를 따른다.

**응답을 로컬에 캐시한다.** 채용 공고, 기업 프로필, 이벤트 데이터는 초 단위로 바뀌지 않는다. API 응답을 애플리케이션 캐시에 저장하고, 매 요청마다 실시간 API를 호출하는 대신 사용자에게 직접 제공한다. 활용 사례에 맞는 주기로 캐시를 갱신한다. 트래픽이 많은 앱은 몇 분마다, 시급하지 않은 데이터는 시간 단위로.

**대량 조회 대신 페이지네이션을 사용한다.** 모든 레코드를 한 번에 가져오기 위해 큰 페이지 크기를 요청하지 않는다. `per_page` 파라미터로 페이지당 적정 수의 결과를 가져오고, 사용자가 실제로 요청할 때만 추가 페이지를 로드한다.

**요청을 중복 제거한다.** 애플리케이션의 여러 부분이 동시에 같은 리소스를 요청할 수 있다면(예: 트래픽 급증 시), 요청 병합 패턴을 사용해 동시 호출이 진행 중인 하나의 HTTP 요청을 공유하게 한다.

**사용량을 모니터링한다.** 응답의 `X-RateLimit-Remaining` 헤더를 확인하고, 정의한 임계값 아래로 떨어지면 경고를 로깅한다. [developers.rocketpunch.com](https://developers.rocketpunch.com)의 개발자 콘솔에도 앱 키의 집계 사용량 통계가 표시된다.

**선제적으로 백오프한다.** `X-RateLimit-Remaining`이 낮은 값으로 떨어지면 `429` 응답을 기다리기보다 자발적으로 요청 속도를 늦춘다. 이렇게 하면 앱이 반응성을 유지하고 사용자에게 서비스 공백이 생기는 것을 피한다.

<Tip>
  읽기 위주 애플리케이션의 경우, 채용 공고와 기업 데이터를 로컬에 캐시하고 정해진 간격(예: 10분마다)으로 갱신하는 것을 고려한다. 이 패턴은 매 사용자 페이지 로드마다 실시간 요청을 보내는 것에 비해 API 호출량을 한 자릿수 배수로 줄일 수 있다.
</Tip>
